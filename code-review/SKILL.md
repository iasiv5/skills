# Code Review Skill

## 审查流程

### Step 1: 理解上下文
**输入**：用户提供的代码 / PR / commit diff + 上下文描述
**输出**：审查范围摘要（语言、框架、变更范围、审查目标）

读取代码后，列出：
1. **代码目的**：这段代码实现什么功能
2. **变更范围**：修改了哪些部分、为什么
3. **业务背景**：相关需求或用户故事
4. **技术栈**：语言、框架、工具
5. **语言/领域检测**：根据文件扩展名和代码内容判断语言与领域

🛑 **STOP — 上下文不足？**
| 条件 | 动作 |
|------|------|
| 代码片段不完整（缺少关键函数/文件） | 向用户请求补充，标注缺失范围后再继续 |
| 无法判断语言/框架 | 向用户确认技术栈 |
| 无明确的审查目标 | 默认进行全维度审查（功能→安全→性能→质量→测试→文档→架构） |

### Step 1.5: 语言/领域专项激活
根据 Step 1 检测到的语言和领域，激活对应的专项检查清单：

| 语言/领域 | 专项检查项 |
|-----------|-----------|
| **C** | 指针安全（野指针、悬垂指针）、内存管理（malloc/free 配对、buffer overflow、use-after-free、double-free）、undefined behavior、`volatile` 正确使用、栈溢出风险、静态分析工具推荐（Cppcheck、Clang-Tidy、splint） |
| **C++** | RAII 合规、智能指针使用（避免裸 new/delete）、移动语义/完美转发、rule-of-five/zero、内存对齐、UB（未定义行为）、ABI 兼容性、Clang-Tidy/Cppcheck |
| **嵌入式通用** | 中断安全（ISR 中禁止阻塞调用）、`volatile` 与 MMIO 正确使用、硬件寄存器访问（原子性、位操作）、内存约束（栈大小、堆碎片）、bare-metal 最佳实践（无 OS 环境下的并发）、看门狗喂狗、功耗关注点 |
| **BIOS/UEFI** | ACPI 表结构验证、SMBIOS Type 规范、UEFI Protocol 接口规范、硬件初始化顺序（PCIe 枚举、内存训练）、复位向量处理、SEC/PEI/DXE/TSL 阶段边界 |
| **BMC** | IPMI 命令格式与完成码处理、传感器阈值与读数校验、Watchdog 配置与超时处理、Redfish API Schema 合规、硬件监控轮询策略、机箱管理（风扇/电源/温度）|
| **Python** | 类型安全、GIL 影响、依赖安全、duck typing 边界 |
| **Java/Go/Rust** | 各语言惯用模式和常见陷阱 |

如果未匹配到任何专项，使用通用审查清单（参考 `resources/REVIEW-CHECKLIST.md`）。

🔴 **STOP — 领域不熟悉？**
| 条件 | 动作 |
|------|------|
| 代码涉及不熟悉的硬件架构/芯片手册 | 坦诚说明局限，审查逻辑和代码质量，对硬件特定行为标注"需硬件专家确认" |
| 代码依赖特定的 SDK/BSP 且无法获取文档 | 标注受审查影响范围，跳过 SDK 内部逻辑 |

### Step 2: 结构化审查
**输入**：审查范围摘要 + 激活的专项清单
**输出**：包含 🔴/🟡/🟢 分级的发现列表

按优先级逐项检查，参考 `resources/REVIEW-CHECKLIST.md` 逐项过检：

| 优先级 | 维度 | 说明 |
|--------|------|------|
| 🔴 必须修复 | 功能正确性、安全性、严重 Bug | 代码未实现预期功能 / 存在注入漏洞 / 逻辑导致崩溃 |
| 🟡 强烈建议 | 性能、测试覆盖、错误处理、架构 | N+1 查询 / 缺少单元测试 / 吞掉异常 / 循环依赖 |
| 🟢 建议 | 代码质量、文档、最佳实践 | 命名不清 / 缺少注释 / 风格不一致 |

要求：
- 每个发现标注优先级标记 + 所在行号/函数名 + 影响 + 修复示例代码
- 🔴 项必须附完整修复示例或明确替代方案
- 不少于 1 条 👍 优点（即使代码质量低，也要找到值得肯定之处）

🛑 **STOP — 审查范围失控？**
| 条件 | 动作 |
|------|------|
| PR/diff 超过 400 行变更 | 向用户建议拆分审查范围（按模块/功能分批），或仅审查核心逻辑路径 |
| 单次审查反馈超过 30 条 | 分两批输出：先输出所有 🔴 + 🟡，再输出 🟢 |
| 代码无法编译运行 | 明确标注"审查基于静态分析，运行时行为未验证"，优先修复编译错误 |

### Step 3: 输出审查报告
**输入**：Step 2 的发现列表
**输出**：结构化审查报告（Markdown）

输出格式：

```raw
## 代码审查报告 → [文件名/模块名]

### 总体评价
[一行总结：如"功能正确但存在 2 个安全漏洞和 3 个性能问题"]

### 🔴 必须修复（N 项）
1. **[行 L42]** buffer overflow：memcpy 目标缓冲区仅 64 字节，源数据可达 256 字节
   - 修复：`memcpy(buf, src, min(src_len, sizeof(buf)))`
2. ...

### 🟡 强烈建议（N 项）
1. **[行 L78-82]** N+1 查询：循环内逐条查询数据库
   - 修复：使用 JOIN 预加载
2. ...

### 🟢 建议（N 项）
1. **[行 L15]** 函数名 `doStuff` 改为 `processSensorData`
2. ...

### 👍 优点
- 错误处理覆盖全面
- ...

### 语言/领域专项备注
- [C] 已检查内存管理和 UB；指针算术部分建议对照 MISRA-C:2012 Rule 18.x
- ...

### 下一步
1. 先修复所有 🔴 项
2. 再处理 🟡 项
3. 提交后再次审查
```

参考 `resources/FEEDBACK-GUIDELINES.md` 撰写反馈措辞。

## 禁止事项（Blacklist）

| # | 禁止行为 | 原因 |
|---|---------|------|
| B1 | 只写 "LGTM" / "看起来没问题" 就结束 | 未进行实质审查，可能遗漏严重问题 |
| B2 | 只批评不给修复方向 | 不建设性，浪费作者时间；🔴 项必须附修复代码 |
| B3 | 使用讽刺、居高临下的措辞 | 违反专业代码审查准则；参考 `FEEDBACK-GUIDELINES.md` |
| B4 | 跳过安全检查 | 即使代码看似非安全敏感，也必须检查输入验证和数据处理 |
| B5 | 照抄代码做"格式化审查" | 不要仅做格式调整就当作审查结果，必须关注逻辑和架构 |
| B6 | 对不熟悉的领域假装权威 | 坦诚标注审查边界（"需硬件专家确认"），而非臆测硬件行为 |
| B7 | 忽略用户的特定审查请求 | 如果用户只要求安全审查，不要强加性能/测试建议；尊重审查范围 |
